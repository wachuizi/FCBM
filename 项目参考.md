# 项目参考（FCBM）

本文档用于你后续“改动模块/做创新”时快速定位：

- 代码入口在哪里
- 数据与特征如何准备、缓存
- FCBM 的两阶段训练（x→c 与 c→y / hypernetwork）在项目中的落点
- 各模块之间的依赖关系、可插拔点

说明：本仓库整体偏“研究代码风格”，许多关键逻辑集中在 `main_vlm.py`、`data/`、`models/`、`utils/` 四处。配置使用 `configs/*.yaml`（OmegaConf）。

---

## 1. 顶层结构概览

仓库顶层（与 `README.md` 中 Directory 基本一致）：

- `main_vlm.py`
  - 训练/测试主入口（Lightning `Trainer`），统一调度两阶段流程。
- `requirements.txt`
  - 依赖列表（PyTorch / PyTorch Lightning / omegaconf 等）。
- `configs/`
  - 各数据集实验配置（如 `cifar10.yaml`、`cub.yaml` 等）。
- `dataset/`
  - 概念集合与（可能的）数据集相关资源。
  - 重点：`dataset/concept_sets/*_filtered.txt`。
- `data/`
  - `LightningDataModule`：负责准备特征、缓存、并构造 dataloader。
- `models/`
  - `LightningModule`：概念学习模块（x→c）与 hypernetwork/分类模块（c→y）。
- `utils/`
  - 训练回调、特征提取/保存、数据工具、稀疏化（sparsemax）等。
- `clip/`
  - CLIP 相关实现（本地化的 CLIP 代码与 tokenizer）。
- `images/`
  - README 用图。
- `saves/`（运行后生成，配置中默认）
  - 缓存的激活、clip 特征、文本特征等（由 `utils.save_activations` 等写入）。
- `logs/`（运行后生成，配置中默认）
  - PyTorch Lightning 日志与 checkpoint。

你做创新时，通常会改动：

- 概念构造/筛选逻辑（`dataset/concept_sets`、`data/data_interface.py`）
- 模型结构（`models/vlm_fcbm.py`）
- 稀疏化/归一化/损失（`utils/sparsemax.py`、`models/*`）
- 训练策略/回调（`utils/*`、`main_vlm.py`）

## 2. 运行入口：`main_vlm.py`

核心流程在 `train(args)`：

1. 读取配置
   - `config = OmegaConf.load(f"configs/{args.d}.yaml")`
   - 配置分 `base / data / model` 三段。
2. 设置随机种子
   - `pl.seed_everything(seed)`
3. 阶段 A：概念学习（x→c）
   - 构造数据：`data_module = VLMDInterface(config.data)`
     - 这一行会触发数据预处理与特征准备（见下文）。
   - 构造模型：`model = FCBMConceptInterface(config.model, text_features)`
   - Lightning 训练/测试：
     - `concept_mode in ['train','both']` → `trainer.fit(...)`
     - `concept_mode in ['test','both']` → `trainer.test(...)`
4. 阶段 B：标签学习 / hypernetwork（c→y）
   - `data_module = VLMDHyperInterface(config.data)`
   - `hyper_model = FCBMHyperInterface(config.model, text_features)`
   - `hyper_mode` 控制训练/测试，与概念阶段类似。

注意点：

- 两阶段共用同一份 `config.model`，但 datamodule 不同。
- `ckpt_path`：既可从配置显式给定，也可以自动从最新 `logs/.../version_*` 中找 `best-epoch=*.ckpt`。
- 训练设备硬编码为 `accelerator="gpu"`；数据模块内部也默认 `cuda:0`（见 `data/data_interface.py`）。这对无 GPU 环境不友好，创新时如果要支持 CPU/多卡，需要同步改入口与 datamodule。

## 3. 配置系统：`configs/*.yaml`

以 `configs/cifar10.yaml` 为例，结构：

### 3.1 `base`：实验控制

- `dataset`: 数据集名（如 `cifar10`、`cub`）。
- `clip_name`: CLIP 模型名（如 `ViT-L/14`）。
- `backbone`: 目标骨干命名（如 `clip_ViT-L/14`，用于提取目标层特征）。
- `concept_mode` / `hyper_mode`: `train/test/both/none`。
- `log_dir`: Lightning 输出根目录（默认 `logs/`）。
- `ckpt_path`: 指定 checkpoint；`None` 表示不指定，让脚本自动找。
- `proj_batch_size` / `hyper_batch_size`: 两阶段 dataloader 的 batch。

### 3.2 `data`：特征与缓存

- `save_dir`: 特征缓存目录（默认 `saves/`）。
- `feature_layer`: 提取目标网络的层名（如 `layer4`）。
- `activation_batch_size`: 抽取特征/激活时的 batch。
- `clip_cutoff`: 概念过滤阈值（在概念筛选处使用）。

### 3.3 `model`：FCBM 超参

- `sparse`: 是否启用稀疏化（通常配合 sparsemax）。
- `concept_lr` / `hyper_lr`: 两阶段学习率。
- `decay_rate`、`act_thred` 等：与激活/筛选/稀疏有关。
- `backbone_config.target`: `models.vlm_fcbm.VLMConceptFCBM`
- `hyper_config.target`: `models.vlm_fcbm.VLMHyperFCBM`

可插拔点：

- 通过替换 `backbone_config.target` / `hyper_config.target` 指向新的类，可以在最少改入口的情况下，接入自己的模型实现（只要接口契合 `models/model_interface.py` 中的 LightningModule）。

## 4. 数据与特征：`data/data_interface.py`

该文件定义两个 datamodule（至少）：

- `VLMDInterface`：概念学习阶段使用
- `VLMDHyperInterface`：hyper/分类阶段使用

### 4.1 概念集合（concept set）来源

在 `VLMDInterface.preprocess()` 里，概念集合路径默认：

- `dataset/concept_sets/{dataset}_filtered.txt`

同时也预留了其它概念来源（注释掉的行）：

- `deepseek_{dataset}_filtered.txt`
- `gpt4o_{dataset}_filtered.txt`
- `expert_{dataset}_filtered.txt`

这意味着：概念层面的创新（例如：概念生成、去重、分组、层级化）基本都可以从这里切入。

### 4.2 类别标签文件

类别名文件通过 `utils.LABEL_FILES` 映射拿到：

- `cls_file = LABEL_FILES[self.dataset.split("_")[-1]]`

### 4.3 特征/激活缓存的生成与读取

`preprocess()` 会对 train/val（这里用 `d_probe` 命名）各跑一遍：

1. 调用 `utils.save_activations(...)`
   - 输入：`clip_name`、`target_name`（backbone）、`target_layers`、`d_probe`、`concept_set`、`pool_mode` 等。
   - 输出：一系列 `.pt` 文件（目标特征、图像特征、文本特征等）。
2. 通过 `utils.get_save_names(...)` 得到各类保存路径
3. `torch.load(...)` 把特征读回内存
4. 用 CLIP 相似度构造 concept 相关特征：
   - `image_features @ text_features.T`

### 4.4 概念过滤（clip_cutoff）

过滤逻辑大意：

- 计算每个 concept 维度上 top-k 响应的均值（代码中 `k=5`）
- 对低于 `clip_cutoff` 的 concept 做剔除/重写 concept 文件（具体实现位置在被截断部分，建议你打开文件继续向下看）

创新提示：

- 如果你要做“动态概念集”（训练时可变）或“按类/按簇概念集”，这里是第一修改点。
- 当前做法是离线缓存 + 过滤；如果你要端到端学习概念文本嵌入/提示词，也需要在 `save_activations` 或其上游动手。

## 5. 模型接口与两阶段模块：`models/`

### 5.1 `models/model_interface.py`

该文件通常提供 LightningModule 的封装接口（如 `FCBMConceptInterface`、`FCBMHyperInterface`），把：

- 网络本体（`VLMConceptFCBM` / `VLMHyperFCBM`）
- 损失计算
- 训练/验证/测试 step
- optimizer/scheduler

绑定到 Lightning 的标准生命周期中。

建议你创新时先明确：

- 你要改的是“网络结构”（`vlm_fcbm.py`）还是“训练目标/优化流程”（`model_interface.py`）。

### 5.2 `models/vlm_fcbm.py`

这里通常是核心模型实现，按配置的 `backbone_config.target` 与 `hyper_config.target` 被实例化。

从 README 的 Pipeline 描述推断（并与 `main_vlm.py` 调度对应）：

- 阶段 A（x→c）：学习从图像/目标特征到概念空间的映射
- 阶段 B（c→y）：使用 hypernetwork，根据文本特征生成权重，对概念进行稀疏加权并预测标签

创新落点（最常见）：

- hypernetwork 结构（文本特征 → 权重生成）
- 稀疏化方式（sparsemax / entmax / L0 近似等）
- 概念瓶颈的约束（互信息、分组稀疏、层次稀疏等）

## 6. 工具层：`utils/`

你会高频用到的几类工具：

### 6.1 特征与数据工具

- `utils/vlm_data_utils.py`
  - 数据集加载、类别文件、概念文件、以及与 `d_probe`（如 `cifar10_train`）相关的解析逻辑。
- `utils/dataloader.py`
  - 自定义 DataLoader（如 `MultiEpochsDataLoader`）或采样策略（加速训练/避免重建迭代器）。
- `utils/vlm_utils.py`
  - `save_activations`、`get_save_names` 等“把模型输出写入磁盘”的核心函数（`data_interface.py` 会调用）。

### 6.2 训练回调

- `utils/base_utils.py`（或 `utils/__init__.py` 导出的封装）
  - `load_callbacks` / `load_hyper_callbacks`
  - 通常包含 early stopping、checkpointing、learning rate monitor 等。

### 6.3 稀疏化：`utils/sparsemax.py`

项目使用 `SparsemaxFunction` 自定义 autograd：

- `sparsemax(X, dim=-1, k=None, temperature=1.0)`
- 内部计算阈值 `tau` 与 support size，然后 `clamp(X - tau, min=0)`

这块是“可解释性/稀疏性”创新的核心基础设施之一：

- 你可以替换/扩展为 entmax、温度退火、组稀疏等
- 也可以改 backward 的温度梯度策略（目前实现了 `grad_temp`）

## 7. 端到端数据流（建议你画在纸上的那条线）

用一条端到端链路描述本项目（从配置到预测）：

1. `python main_vlm.py -d cifar10`
2. 读取 `configs/cifar10.yaml`
3. `VLMDInterface.preprocess()`
   - 读取 concept set + class names
   - 调 `save_activations` 抽取/缓存 train & val 的特征
   - 读回特征并构造 CLIP 相似度概念特征
4. `FCBMConceptInterface`（阶段 A）
   - 输入：目标特征（可能是 backbone 某层激活）
   - 输出：概念预测/概念向量
5. `VLMDHyperInterface` + `FCBMHyperInterface`（阶段 B）
   - 输入：概念表征 + 文本特征（concept text features）
   - hypernetwork 生成权重 → sparsemax 稀疏化 → 输出分类 logits
6. Lightning 写日志到 `logs/`，特征缓存到 `saves/`

## 8. 创新/改动模块的“最小切入点”清单

按影响范围从小到大排列（便于你迭代）：

1. 只改概念集合文件
   - 直接替换 `dataset/concept_sets/{dataset}_filtered.txt`
   - 或在 `data/data_interface.py` 切换到 deepseek/gpt4o/expert 文件
2. 改概念过滤策略
   - 改 `VLMDInterface.preprocess()` 的 clip 相似度统计、top-k、阈值逻辑
3. 改稀疏化算子
   - 改 `utils/sparsemax.py` 或在模型中替换调用
4. 改 hypernetwork
   - 改 `models/vlm_fcbm.py` 中“文本→权重”的映射
5. 改两阶段训练范式
   - 改 `main_vlm.py` 的调度、共享/冻结策略、以及 checkpoint 逻辑
